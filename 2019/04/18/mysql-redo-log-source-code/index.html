本文基于MySQL 5.7分支的源代码，关于其他版本请自行查阅相关代码

本文写于2019/04/18。

Overview

Note：本文假定读者已对MySQL InnoDB所实现的WAL（Write-Ahead-Log）机制、checkpoint、redo log是什么有基本的了解。

由于MySQL InnoDB里基于WAL的理论实现了redo log和checkpoint，所以在一定前提下能保证数据不丢失，当数据库进程或操作系统或物理机崩溃之后还能从redo log中把未刷脏的数据找回，恢复数据库。

这一整套写redo log、checkpoint、崩溃恢复的流程里，最关键的元素之一莫过于redo log本身和LSN（Log Sequence Number）的概念。但是到底redo log的底层结构是怎样的呢？LSN为什么这么关键呢？我尝试从各种讲InnoDB的书或网上的知名博客或MySQL的官方文档中找答案，但我找到的资料没有一个能完全把整个redo log的结构及相关的checkpoint + 恢复流程讲得简单清晰，且都没有提供一些让人易懂的例子。

包括 MySQL官方文档，讲得比较粗略、生涩；京东架构师王伟的redo log浅析，图文并茂，读起来很舒服，但是有一些变量含糊带过，但像走到了宝藏门前却发现钥匙不见了；淘宝数据库内核月报-2015/05 还有 淘宝数据库内核月报-2017/09，排版和内容长度不太友好，而且其中一篇引用了大量源码里的各个变量，但却不完整引用源码，新手读着读着有点晕。

所以，我决定结合着上述几篇文章，然后直接去看MySQL 5.7分支的源码。然后根据我对源码的理解，做一些redo log和checkpoint的例子图（请停止互相复制粘贴水博客），帮助后来的读者理解redo log的结构和checkpoint相关的内容。

概念

虽然假定本文的读者对相关的概念都有基本的了解，此处还是提一下重要的概念：

  • redo log

    对内存中的数据页作修改前，基于WAL机制，要先写日志后修改值，这里的日志指的就是redo log。同时，为了不让Disk I/O拖垮整个系统的性能，所以一条又一条的redo log不能直接写磁盘，而是要先写到redo log buffer（内存）里，然后某个时间点让某个线程把redo log buffer里的redo log批量刷到磁盘。

  • LSN

    对内存中的数据页修改前，要写redo log，每条redo log都会有自己的LSN（Log Sequence Number），一个像唯一id似的递增的值。

    然后真正去修改数据页时，也会把该LSN记录在页上，代表着该页被人修改过，已经变成脏页。那怎么知道修改对应产生的log是哪条呢，就是LSN标识的那条。

    同时，在checkpoint的时候，系统也会把已经完成checkpoint的LSN记录下来，那么我们就知道了“哦，在checkpoint LSN前的所有修改，都已经完好无损地刷到磁盘了”。而在checkpoint LSN后的LSN，则只是写好了日志到redo log里，数据页还没真正刷盘。

    不过为了严谨，还是郑重声明一下，其实LSN并不是我们平常所见的1、2、3 …… N这样的自增值，严格意义上LSN = 从系统刚开始到现在已经向日志系统写入的总字节数。对于每条redo log而言，它是唯一的，也是递增的。

    另外，严格意义上来说，其实不存在“一条redo log”这样的说法，因为一条log的基本单位是一条mini transaction（mtr）log，而redo log只是一种日志的笼统说法，是一种概念。具体可参考下图：

所以，我们需要研究以下概念：

  • mini transaction是什么结构，每条mini trx log里有什么
  • redo log buffer是什么结构，在内存里是怎么存一条一条mini trx log的
  • redo log file (on disk)是什么结构，在磁盘上是怎么存一条一条mini trx log的

Mini Transaction Log

上面说过，每个事务里的每条SQL语句，其实最终在内部都会被分解成mini transaction来执行。一个mini transaction对应着一条mini transaction log，那我们很自然地会觉得，在源码里肯定会有一个Class或者Struct专门描述mini trx log的。

然而并没有专门的某个Class或Struct叫什么MiniTrxLog的，而是用最硬核的方式，全部借助指针裸操作byte数组，规定好数组里第几个byte是表示什么。

先上我总结的结构图，再看源码：

  • Mini Trx Log Type，占了1 byte，表示这条mini trx log对应的mini trx是什么类型的，是写入还是删除还是其他别的什么操作。
  • Space id，占了4 bytes，表示这条mini trx log对应的mini trx是在哪个table space里操作的。
  • Page No，占了4 bytes，表示这条mini trx log对应的mini trx改了哪个数据页。
  • Record Payload，变长字段，表示这条mini trx log真正的内容，对应不同log type记录的内容也不一样。例如一条insert的操作就要记录insert了什么列，而delete的操作就要记录delete了什么列，等等。

Redo Log Buffer

研究完mini trx log的格式，接下来看看存放每条mini trx log的地方——Redo Log Buffer。

不难想象，既然每一条mini trx log都只是一串byte组成的字符串，那么很有可能存放log的地方也只是一个普通的byte buffer（数组），只不过这个数组容量要尽量大一点才行。如下图所示：

其实以上说法也对也不对，来看看源码好了，请查看 mysql-server/storage/innobase/include/log0log.h 的第624行，由于篇幅原因本文已去除部分代码注释：

从源码中看到跟redo log buffer最直接关联的就是：

  • ulint buf_size

    redo log buffer的总大小。

  • byte* buf

    当前正在使用的redo log buffer本体，一个byte指针指向的byte数组。（大小不超过buf_size）

  • byte* buf_ptr

    一个隐性的buffer，大小为2 * buf_size，写log的时候不直接对该buffer操作。在并发操作的场景下，buf指针要么指向buf_ptr的前1/2，要么指向后1/2，轮流交替。

  • ulint buf_free

    该变量并不是指buffer里还剩余多少byte可用，而是指buffer里剩余空间的起始点的offset，也就是接下来要写log的起始位置。

各个变量之间的关系如上图所示，从图里可以清楚看到为什么要将隐性buffer（*buf_ptr）设成正常使用的buffer（*buf）的两倍，通过这样设置就可以在flush时继续写入log，不至于因为Disk I/O而阻塞log的写入。

P.S. 这种双冗余的做法在JVM的GC算法（Copying算法）里也有类似的体现，设置两个survivor区域，每次只用其中一个，轮流切换。

Redo Log File

了解了mini trx log和内存中redo log buffer的结构后，是时候来看看log被刷到磁盘后是怎么存的了，指的就是磁盘上的Redo Log File的结构。

根据 MySQL官方文档 和5.7分支的源码，每个Redo Log File的结构图如下：

可以看出，MySQL在redo log的存储方面用的是定长格式（Fixed-Length），每个基本块的大小都是固定的512 bytes。

Note：整个系统可以有不止一个1个Redo Log File。如上图所示，每个Redo Log File的头部都是固定占了2KB（4 * 512 bytes）大小的，不过只有第一个Redo Log File的头部2KB才真正含有有意义的数据。

接下来详细解释一下每个结构里各个字段的含义。

  1. Redo Log File

    • Log File Header（512 bytes）：记录了这个日志文件的meta data，例如是谁创建的，文件里记录的第一条日志的LSN是多少……
    • Checkpoint Block #1（512 bytes）：记录了Checkpoint信息的块，每当系统执行完Checkpoint操作后都要把相应的Checkpoint数据写入此处。
    • Unused（512 bytes）：无实际数据，仅限占位。
    • Checkpoint Block #2（512 bytes）：功能同#1一样，每发生一次Checkpoint会轮流写#1和#2块。（又一个双冗余的设计）
    • N * Log Block：N个真正的日志块，每个块固定512 bytes，记录了每条mini trx log的内容。

  2. Checkpoint Block

    • Checkpoint No (8 bytes)：系统最近一次执行的Checkpoint操作的编号，每次Checkpoint完成后自增，最大取值为2^64 - 1。当Checkpoint No为奇偶数时分别轮流写Checkpoint Block #1和#2。
    • Checkpoint LSN (8 bytes)：系统最近一次执行的Checkpoint操作对应的LSN。
    • Checkpoint Offset (8 bytes)：最近一次执行的Checkpoint对应的LSN在日志文件里的Offset。
    • Buffer Size (8 bytes)：最近一次Checkpoint执行时系统的redo log buffer大小（记录这个并没有太多实际意义）。
    • Padding (480 bytes)：占位，为了填充满512 bytes。

  3. Log Block

    • Log Block No (4 bytes)：该Log Block的编号，相当于全局自增id。
    • Data Len (2 bytes)：该Log Block已经写入了多少bytes，包括头部这固定的12 bytes。
    • First Record Offset (2 bytes)：该Log Block中第一条mini trx log的offset。
    • Log Block Checkpoint No (4 bytes)：该Log Block所属的Checkpoint编号。
    • N * Mini Trx Log：N条mini trx log，直到本Log Block装不下为止。
    • Checksum (4 bytes)：对本Log Block里数据的总校验和，以防数据出错。

Log Block结构里的怪事

读到这里，感觉我们把该分析的结构都分析完了，该看的必要的源码也都看了，redo log以及redo log相关的结构都比较清晰了，本文应该可以结束了吧？

但是，细心的读者此时应该会发现，上述解析里有一些关于Checkpoint比较奇怪的点：

为什么Log Block结构里会有一个Log Block Checkpoint No (4 bytes)？？？

写redo log的时候怎么会知道这个log对应的数据脏页啥时候会被Checkpoint刷盘？？？

在大多情况下，redo log的写入会明显快于Checkpoint的发生，怎么可能还没发生Checkpoint就知道自己对应的是哪次Checkpoint，未卜先知？？？

带着问题去看别人的博客，MySQL官方文档 里只是说Checkpoint number of a next checkpoint write. Set by the log writer thread just before a write starts for the block.；京东架构师王伟的redo log浅析里面简单地说它是“该block所处在的checkpoint no”；淘宝数据库内核月报-2017/09 里直接完全忽略了这个字段，提都没提。

那到底为什么在写redo log时能够“未卜先知”，知道该redo log对应哪一次Checkpoint？还得仔细看看源码。在上述log_write_low()函数里有这么一段：

可以看到，当一个log block里的512 bytes都被写满后，就会执行log_block_set_checkpoint_no()，用的是log_sys->next_checkpoint_no。

也就是说，整个日志系统会维护一个全局的Checkpoint No，每发生一次Checkpoint这个Checkpoint No就会自增。在上一次Checkpoint C0结束后，到下一次Checkpoint C1发生前的这一整段时间里，所有的redo log都被归属于C1。如下图所示：

这时又引申出一个新的问题：怎么保证下一次Checkpoint时，能把归属于它的所有redo log对应的数据脏页都刷到磁盘上？会不会出现下图这样的情况？

假如像上图所示，在某次Checkpoint C1发生时，刷到磁盘的脏页只包含到LSN = X + 3000的那一页，并没有包含最新的一页（LSN = X + 8000）。在这种情况下，LSN = X + 8000的那个redo log block早就被归属于Checkpoint C1，却没有在Checkpoint C1时把该log对应的脏页刷盘，岂不是出大错了？

这个问题的根源其实就在于“脏页刷盘的逻辑”，到底刷脏时会不会保证把最新的一个脏页刷到磁盘？如果能够保证Checkpoint每次都把当前LSN最大的一个脏页刷盘，那么提前给每个redo log block指定其归属的Checkpoint No就不会有问题，因为当下次Checkpoint C1发生时会保证把C0到C1间的所有脏页都刷盘。

带着问题，我们去看源码 mysql-server/storage/innobase/include/buf0dblwr.h 里第331行：

看来我们的猜想没错。当执行Checkpoint时，总会尝试把最新的脏页（LSN_MAX）刷盘。这样就能保证每个Redo Log Block里的Checkpoint No不会出错了，每个Redo Log Block都属于下一次Checkpoint~

总结

Redo Log在MySQL里是一项非常重要的概念，但是网上很多讲Redo Log结构的文章都只是草草描述，或者照抄《MySQL InnoDB存储引擎》里的内容；又或者是另一个极端，复制粘贴讲了一大堆源码，却没有太多对应的例子和示意图，读起来非常不友好。所以我在研究Redo Log的时候就决定写下这篇博客，希望能帮助对此感兴趣的读者更好地理解Redo Log的结构，从而更好地理解Redo Log + Checkpoint整个机制。
